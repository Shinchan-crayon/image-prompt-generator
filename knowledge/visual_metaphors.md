# 视觉隐喻关系库

## 何时读取

核心观点已经提炼为一种关系、变化或冲突，但还缺少可拍摄的表达时读取。本文件负责把
抽象观点映射为现实场景、主体和动作，不负责决定文章分类、运行模式或最终 Prompt。

## 快速决策

先识别观点的主关系，再读取对应条目：

| 主关系 | 优先观察对象 | 视觉动作 |
|---|---|---|
| 能力边界改变 | 同一任务、同一操作者 | 原本失败的步骤被稳定完成 |
| 自主执行 | 任务链、检查点、人工批准 | 系统推进，人只在关键点确认 |
| 资源瓶颈 | 队列、槽位、设备负载 | 有限资源被分配或等待 |
| 竞争 | 同一入口、货架或客户任务 | 多方争夺同一稀缺位置 |
| 信任与安全 | 权限边界、检查记录、隔离区 | 人核对、阻断或放行 |
| 增长 | 订单、产能、服务空间 | 处理能力随需求有序扩展 |
| 平台关系 | 共享接口、工作台、调度中心 | 多方通过同一规则接入 |
| 协作 | 交接点、共同工件、审核节点 | 不同角色连续修改同一对象 |
| 风险 | 薄弱节点、未确认路径 | 人发现异常并暂停下一步 |
| 学习与判断 | 反馈记录、练习成果、专业资料 | 人依据反馈修正或确认 |
| 规则约束 | 检查清单、边界线、审批台 | 行为在规则前被验证或改道 |

选择后必须能填满：

`通过【一个主体】在【现实地点】执行【一个动作】，让【关系或变化】可见。`

若需要三个以上主要对象或两种独立动作，隐喻仍然过宽，应返回核心观点继续收敛。

## 条目字段

每条记录使用同一结构，便于持续扩充和组合：

- **适用关系**：文章中需要被视觉化的关系。
- **现实场景**：现实中可拍摄的地点和物件。
- **主体与动作**：唯一主体及可观察动作。
- **为什么成立**：场景与观点的对应关系。
- **不适用**：应改用其他条目的条件。
- **撞图风险**：最容易退回的陈词滥调。

## 01. AI Agent

- **适用关系**：从单次回答转向多步骤执行、工具调用、任务接力或人工监督。
- **现实场景**：运营人员的真实工作台，一项任务经过日历、表格、邮件和审批记录连续推进。
- **主体与动作**：运营人员只在关键检查点确认，其余重复步骤在同一屏幕或桌面记录中依次完成。
- **为什么成立**：重点是任务连续推进与责任边界，不是把 AI 拟人成另一个员工。
- **不适用**：文章只讨论一次问答质量、模型参数或单项生成能力时，改用“模型能力”。
- **撞图风险**：机器人秘书、数字人分身、多个悬浮应用图标、代码雨。

## 02. 模型能力

- **适用关系**：能力提升、能力边界、稳定性差异、泛化或特定任务表现。
- **现实场景**：研究员或专业用户在同一工作台上，用同一材料重复完成一项明确任务。
- **主体与动作**：同一人物把困难样本交给系统，并核对一次更完整、准确或稳定的结果。
- **为什么成立**：固定任务和操作者后，画面只让“可完成范围或完成质量”发生变化。
- **不适用**：文章重点是部署成本、集群吞吐或连续任务执行时，分别改用“GPU / 算力”或“AI Agent”。
- **撞图风险**：发光大脑、参数数字墙、模型擂台、AI 人脸、夸张考试分数。

## 03. GPU / 算力

- **适用关系**：有限计算资源、任务排队、吞吐扩张、能耗约束或算力调度。
- **现实场景**：数据中心运维通道、计算任务调度台，或带有限插槽的桌面实物演示。
- **主体与动作**：运维人员把等待队列分配到有限计算槽位，部分任务继续排队。
- **为什么成立**：算力被表现为受容量、时间和能耗约束的服务能力，而不是一块神奇芯片。
- **不适用**：文章重点是制程、封装、供应链或芯片制造时，应使用半导体生产现场。
- **撞图风险**：巨大显卡、芯片悬浮城市、无限服务器走廊、蓝色电路光、算力洪流。

## 04. 商业竞争

- **适用关系**：争夺客户、渠道、定价权、供应位置、速度或差异化焦点迁移。
- **现实场景**：同一货架、采购评审桌、应用入口或供应排期板。
- **主体与动作**：采购者把有限席位或订单移向更符合关键条件的一方。
- **为什么成立**：竞争通过同一稀缺资源下的真实选择被看见，而不是把公司拟人成战士。
- **不适用**：文章只分析单一公司的内部能力，没有直接比较或资源争夺时，改用组织或产品场景。
- **撞图风险**：拳击台、赛马、棋盘、Logo 对撞、握手、胜利者站上领奖台。

## 05. 安全

- **适用关系**：访问控制、验证、隔离、审计、可追溯性或有效防护。
- **现实场景**：企业权限审核台、实验室隔离区、运维检查现场或医疗资料访问终端。
- **主体与动作**：负责人核对访问记录，将异常请求隔离，并放行已验证路径。
- **为什么成立**：安全是可检查的边界与持续动作，不是“绝对不会出事”的符号。
- **不适用**：文章只讨论潜在损害、概率或尚未建立防护时，改用“风险”。
- **撞图风险**：巨大锁头、盾牌、绿色安全墙、黑客兜帽、红色代码雨。

## 06. 增长

- **适用关系**：需求增加、能力积累、产能扩张、留存或可持续规模化。
- **现实场景**：小型仓库、服务柜台、生产线、教室或逐步扩展的工作空间。
- **主体与动作**：工作人员增加一个真实处理环节，使更多订单或用户被有序承接。
- **为什么成立**：增长被表现为需求与承载能力一起变化，并保留成本和操作约束。
- **不适用**：文章只有融资金额、估值或短期流量，没有业务承接证据时，不应暗示持续增长。
- **撞图风险**：火箭、上升箭头、幼苗、金币堆、无限复制的人群或产品。

## 07. 平台

- **适用关系**：共享入口、规则制定、生态接入、分发、依赖或网络效应。
- **现实场景**：共享工作台、物流分拨台、统一接口柜台或多方使用的调度中心。
- **主体与动作**：不同参与者把各自工具接入同一标准入口，平台方调整一条共同规则。
- **为什么成立**：平台价值来自连接与规则，不来自把许多产品围成一圈。
- **不适用**：文章只是多个产品功能集合，没有共同接口、分发或治理关系时，不应使用平台隐喻。
- **撞图风险**：Logo 星系、中心发光球、应用图标环、万能控制台、无来源连接线。

## 08. 协作

- **适用关系**：角色分工、连续交接、共同编辑、反馈闭环或人机分工。
- **现实场景**：编辑桌、设计评审墙、工程检修台、医疗会诊桌或共享文件工作区。
- **主体与动作**：两名角色围绕同一工件连续标注、修改和确认，焦点始终落在共同对象。
- **为什么成立**：协作由共享目标、交接顺序和责任动作构成，而不是“大家站在一起”。
- **不适用**：文章重点是单人效率提升或纯自动执行时，应保持一个操作者。
- **撞图风险**：商务握手、围桌微笑、人与机器人碰拳、拼图块、多人各看各的屏幕。

## 09. 风险

- **适用关系**：潜在失败、暴露面扩大、不可确认路径、单点薄弱或后果不确定。
- **现实场景**：上线前检查台、道路测试现场、权限审核桌、生产巡检点。
- **主体与动作**：负责人在异常迹象前暂停下一步，并沿记录追查薄弱节点。
- **为什么成立**：风险表现为尚未变成事故的可见弱点，以及人对不确定性的审慎响应。
- **不适用**：损害已经发生或系统正在失效时，应使用危机现场；防护已验证有效时改用“安全”。
- **撞图风险**：断崖人物、灾难城市、满屏警报、红色骷髅、把小概率画成既成事故。

## 10. 教育

- **适用关系**：理解、反馈、个性化支持、能力积累、教师判断或学习差异。
- **现实场景**：真实教室、家庭书桌、实验课或教师备课桌。
- **主体与动作**：教师依据学生的练习痕迹调整下一步提示，学生修正同一项任务。
- **为什么成立**：教育价值来自反馈后发生的学习变化，不是设备进入教室本身。
- **不适用**：文章只介绍教育软件功能，没有学习行为或结果证据时，不应暗示教学成效。
- **撞图风险**：学生围着机器人、发光知识树、统一笑脸、毕业帽、屏幕代替教师判断。

## 11. 医疗

- **适用关系**：辅助判断、证据核对、流程提效、责任边界、隐私或患者安全。
- **现实场景**：诊室、影像阅片室、病理工作台、护士站或医院资料审核台。
- **主体与动作**：医生把系统提示与原始检查资料并置核对，再做出确认或退回决定。
- **为什么成立**：技术提供辅助信息，专业人员承担判断，患者尊严和证据链保持可见。
- **不适用**：没有临床证据时，不应画成已治愈、已确诊或完全替代医生。
- **撞图风险**：全息人体、机器人医生、夸张病灶发光、患者隐私暴露、未来医院广告。

## 12. 政策

- **适用关系**：规则改变行为边界、合规成本、实施落差、审批或多方适应。
- **现实场景**：企业合规桌、公共服务窗口、道路标线更新现场或行业检查清单。
- **主体与动作**：执行人员依据新规则修改一个具体流程，并把不符合项退回调整。
- **为什么成立**：政策通过现实执行动作影响行为，不把“文件发布”误写成“效果已经实现”。
- **不适用**：文章只是政治人物或会议新闻，尚未说明规则内容与影响对象时，不应推断实施结果。
- **撞图风险**：法槌、国旗拼贴、巨大印章、文件从天而降、把政策画成单向命令。

## 13. 自动驾驶

- **适用关系**：感知与决策、运营边界、人机接管、道路协商、安全验证或规模部署。
- **现实场景**：真实城市道路、封闭测试场、车内驾驶位或远程运营中心。
- **主体与动作**：安全员在复杂路口观察系统减速并准备接管，或运营员核对一段异常路线。
- **为什么成立**：核心是车辆如何在真实约束下判断，以及人何时监督或接管。
- **不适用**：文章只讨论车载芯片性能时改用“GPU / 算力”；没有证据时不画成完全无人、全城覆盖。
- **撞图风险**：霓虹城市、透明汽车、满路激光线、空无一人的未来道路、睡觉的驾驶员。

## 组合与扩充规则

1. 一张图只选一个主条目；相邻条目只用于检查边界，不叠加两个隐喻。
2. 分类决定默认主体路径、地点和真实物件，主关系决定人的动作或物体状态变化，情绪只调整距离、光线和对比。
3. 同一条目连续使用时，至少更换地点、工件或动作，避免形成固定模板。
4. 新条目必须补齐六个固定字段，并说明与现有条目的区分。
5. 无法现实拍摄时，先改造成桌面实物演示；物体状态足够清楚时无需额外加入手部，不要退回机器人、Logo、代码雨或漂浮 HUD。
