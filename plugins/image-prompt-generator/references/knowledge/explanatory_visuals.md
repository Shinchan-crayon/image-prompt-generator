# 解释型视觉子策略

## 何时读取

只在**正文配图模式**中，当前章节需要解释关系、过程、结构、界面行为或状态变化时读取。
本文件提供正文配图的子策略，不新增“信息图模式”“UI 模式”或任何第三种运行功能。

## 快速选择

按以下顺序选择第一个足够清楚的策略：

1. 现实人物或物体的一个动作能解释清楚：使用**现实场景**。
2. 机制抽象，但可用纸卡、筹码、容器或文件演示：使用**桌面演示**。
3. 章节重点是软件界面中的真实操作：使用**真实 UI**。
4. 读者必须理解步骤先后：使用**流程**。
5. 读者必须看清一个变量如何改变结果：使用**同主体对照**。

一次只选择一个主策略。桌面演示、真实 UI、流程和对照都属于正文配图模式内的
解释方法，不应改变输出模板或运行入口。

## 共同约束

- 只解释当前章节的一个中心问题。
- 一个主体、一个主要动作、一个焦点；概念节点不超过四个。
- 使用位置、顺序、接触、容器、材质或状态表达关系，文字只是辅助。
- 不要求模型生成长段说明、精确数据或未经输入确认的界面内容。
- 解释清晰与视觉冲击冲突时，选择解释清晰。

## 子策略库

### 01. 现实场景

- **适用**：人的选择、责任、体验、学习、医疗判断、道路接管或真实设备操作。
- **结构**：人物或实物在具体地点执行一个可观察动作，关系直接发生在现场。
- **镜头**：中景、肩后视角或操作近景。
- **信息上限**：一个动作、一个结果状态，辅助物件不超过两类。
- **不适用**：关系无法在单一现场被看见，必须依赖多个时间阶段时改用流程或桌面演示。
- **常见错误**：人物摆拍、设备英雄照、用表情代替判断、背景加入无关科技元素。

### 02. 桌面演示

- **适用**：资源分配、限制变化、成本转移、依赖关系、平台接入或抽象机制。
- **结构**：真实桌面、少量纸卡或筹码、一个容器或路径，以及一个清楚的移动、排队、分流或转换状态。
- **镜头**：垂直或轻微斜俯视，保持物件真实比例和清晰层级。
- **信息上限**：三到四个节点；若需要标签，使用短词或不依赖可读文字的形状差异。
- **人物规则**：操作来源、责任或尺度必须可见时才加入手部；物体状态已经清楚时不强制出现人物或手。
- **不适用**：文章核心是人的体验、专业责任或真实环境风险时，不应用桌面道具替代现场。
- **常见错误**：像棋盘、玩具化、均匀铺满、颜色编码过多、把桌面演示写成抽象信息图。

### 03. 真实 UI

- **适用**：章节讨论一次具体软件操作、界面反馈、权限确认、Agent 执行记录或产品体验差异。
- **结构**：真实设备、低噪音界面和一个与观点直接相关的状态变化；需要体现操作或责任时再加入人物或手部动作。
- **镜头**：设备现场近景、肩后视角或屏幕与手同框的近景；屏幕不应成为脱离现实的整页截图。
- **信息上限**：一个主界面、一个关键状态、最多一个辅助面板。
- **文字规则**：只使用输入已确认的短标签；无法可靠生成时，用位置、色块和状态代替。
- **不适用**：界面只是承载工具，核心其实是商业、教育或医疗结果时，应回到现实场景。
- **常见错误**：虚构按钮、随机图表、密集小字、多设备漂浮、把聊天框当成所有 AI 的通用画面。

### 04. 流程

- **适用**：工作流、审批、数据处理、生产、Agent 多步骤执行或服务交付。
- **结构**：一个入口、一条单向路径、三到四个状态、一个关键转换点。
- **镜头**：俯视桌面、真实工作台或轻微轴测；优先用同一工件沿路径变化。
- **信息上限**：通常三步，必要时四步；每步只显示一次状态变化。
- **不适用**：步骤顺序不是章节重点，或一个现场动作已经足以说明关系时，不要使用流程。
- **常见错误**：循环箭头、分支过多、每一步变成独立场景、装饰性连接线、依赖大段文字。

### 05. 同主体对照

- **适用**：前后变化、两种限制机制、产品体验差异、能力边界或政策前后的一个变量。
- **结构**：保持同一主体、视角、尺度、背景和光线，只改变一个被解释变量。
- **镜头**：固定机位的并置、连续状态或同一桌面上的相邻区域。
- **信息上限**：两个状态，一个差异；必要标签只说明变量，不评价好坏。
- **不适用**：两侧需要不同人物、不同地点和不同动作才能成立时，说明它们是两个故事，应重新收敛。
- **常见错误**：情绪化左右分屏、红坏绿好、同时改变多个变量、把对照做成广告前后效果图。

## 关系与策略映射

| 需要解释的关系 | 优先策略 | 可见证据 |
|---|---|---|
| 人做判断，系统辅助 | 现实场景或真实 UI | 人的确认、退回或接管动作 |
| 有限资源如何分配 | 桌面演示 | 数量有限的物件随任务移动 |
| 多步骤如何推进 | 流程 | 同一工件沿单向路径变化 |
| 两种机制差在哪里 | 同主体对照 | 只改变一个限制条件 |
| 软件反馈如何影响行为 | 真实 UI | 操作与界面状态直接相连 |
| 规则如何改变现实流程 | 现实场景或流程 | 检查、阻断、改道或重新提交 |

## 留白与标签

- 正文配图不强制大面积标题安全区，但应保留足够呼吸空间，避免信息贴边。
- 标签应靠近对应物体，不跨越动作路径，不形成第二视觉层。
- 需要三个以上句子才能解释图时，先减少节点或更换策略，不把说明文字塞进画面。
- 若平台需要后期章节标题，可在不损害关系的前提下保留一侧约 20%–30% 的低纹理区域。

## 失败时重选

- **看起来像封面**：降低戏剧光线与象征性，增加真实操作证据。
- **看起来像信息图**：把图标替换为真实物件、界面或同一工件的状态。
- **仍需长文解释**：减少到一个关系，或从流程改为同主体对照。
- **UI 不可信**：缩小界面占比，只保留一个已确认状态，让界面反馈或必要的人类动作承担叙事。
- **对照有两个焦点**：固定主体和环境，只保留一个变化变量。
